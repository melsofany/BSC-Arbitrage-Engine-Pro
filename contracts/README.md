# 🚀 Multi-Path Flash Arbitrage Engine v2.0

هذا العقد الذكي هو النسخة المتقدمة والمحسنة من محرك المراجحة (Arbitrage Engine) الخاص بك. تم تصميمه ليكون **الأكثر مرونة، والأقل استهلاكاً للغاز، والأكثر شفافية** في تتبع الصفقات.

---

## 🌟 الميزات الرئيسية

### 1. 🛣️ دعم المسارات المتعددة (Multi-path)
بدلاً من الاكتفاء بمسار واحد (Token A -> Token B -> Token A)، يمكنك الآن تنفيذ أي عدد من الخطوات المعقدة:
- **Buy Hops:** سلسلة من التبادلات لشراء العملة المستهدفة.
- **Sell Hops:** سلسلة من التبادلات لبيع العملة المستهدفة والعودة للعملة الأساسية (Loan Token).
- يمكنك مثلاً القفز من: `WBNB -> BUSD -> USDT -> CAKE -> WBNB`.

### 2. 🔄 دعم منصات متعددة (Multi-DEX)
العقد يدعم التبديل بين المنصات المختلفة في نفس الصفقة:
- **V2 DEXs:** PancakeSwap V2, BiSwap, ApeSwap, BabySwap, MDEX.
- **V3 DEXs:** Uniswap V3, PancakeSwap V3.
- *لا تحتاج إلى Wrappers إضافية، العقد يتعامل معها داخلياً عبر متغير `dexVersion`.*

### 3. ⛽ تحسين فائق لاستهلاك الغاز (Gas Optimized)
- تم التخلص من الـ Structs الثقيلة في الذاكرة (Memory).
- تم استخدام `calldata` بدلاً من `memory` لتقليل تكلفة استدعاء الدوال.
- تم دمج الواجهات (Interfaces) لتقليل حجم العقد وتكلفة النشر.

### 4. 📊 نظام تتبع شامل (Full Event Logging)
كل خطوة يتم تنفيذها داخل العقد تقوم بإصدار حدث (Event) مفصل، مما يسمح للبوت الخاص بك (Off-chain) بقراءة ما حدث بالضبط، حتى لو فشلت الصفقة.

## 📡 الأحداث (Events) المتاحة للتتبع

هذه قائمة بالأحداث التي يمكنك الاستماع إليها عبر الـ Backend الخاص بك:

### أحداث دورة الحياة (Lifecycle)
- `ArbStarted`: يُطلق عند بدء الصفقة (يحتوي على المبلغ، الديدلاين، وعدد الخطوات).
- `FlashLoanReceived`: يُطلق عند استلام الفلاش لون (يحتوي على المبلغ والسيولة المتاحة).
- `ArbFinished`: يُطلق عند انتهاء الصفقة بنجاح.

### أحداث التبادل (Swaps)
- `SwapExecuted`: **أهم حدث!** يُطلق بعد كل خطوة تبادل (Hop) ويحتوي على:
  - `leg`: 0 (شراء) أو 1 (بيع).
  - `stepIndex`: رقم الخطوة.
  - `dexRouter`: عنوان المنصة المستخدمة.
  - `estimatedOut`: الكمية المتوقعة.
  - `actualOut`: الكمية الفعلية التي تم الحصول عليها.
  - `slippageWei`: الفرق بين المتوقع والفعلي (رقم سالب يعني انزلاق سلبي).

### أحداث الربح والتسوية (Settlement)
- `Settlement`: يُطلق عند تسديد القرض ويحتوي على:
  - `loanAmount`: مبلغ القرض الأساسي.
  - `fee`: رسوم الفلاش لون (مثلاً 0.3%).
  - `netProfit`: الربح الصافي.
  - `profitBps`: الربح كنسبة مئوية (Basis Points).
- `ProfitSent`: يُطلق عند تحويل الربح لمحفظتك.

### أحداث الأخطاء والتصحيح (Debugging)
- `Debug`, `DebugAddr`, `DebugBalance`, `DebugStep`: أحداث تتبع تفصيلية للمساعدة في حل المشاكل.
- `ExecutionFailed`: يُطلق إذا فشلت الصفقة، مع ذكر السبب والخطوة التي فشلت فيها.

## 💻 كيفية بناء الـ Payload (Calldata)

لتنفيذ صفقة، يجب على البوت الخاص بك (Node.js/Python) بناء الـ `ArbParams` وتمريرها لدالة `executeArbitrage`.

### هيكل `SwapHop`
لكل خطوة (Hop)، تحتاج إلى تحديد:
1. `dexRouter`: عنوان الراوتر (مثلاً PancakeSwap V2 Router).
2. `tokenIn`: العملة المدخلة.
3. `tokenOut`: العملة المخرجة.
4. `amountOutMin`: الحد الأدنى المقبول (للحماية من الانزلاق).
5. `fee`: رسوم الحوض (Pool Fee) لمنصات V3 (مثلاً 500 لـ 0.05%). ضع 0 لـ V2.
6. `dexVersion`: 
   - `0`: V2 (PancakeSwap V2, BiSwap, etc.)
   - `1`: Uniswap V3
   - `2`: PancakeSwap V3
7. `quoter`: عنوان الـ Quoter لمنصات V3 (لجلب الأسعار المتوقعة). ضع `address(0)` لـ V2.

### هيكل `ArbParams`
1. `flashPair`: عنوان الزوج (Pair) على PancakeSwap V2 الذي ستأخذ منه الفلاش لون.
2. `loanToken`: العملة التي ستقترضها (مثلاً WBNB).
3. `loanAmount`: مبلغ القرض.
4. `buyHops`: مصفوفة (Array) من `SwapHop` لعمليات الشراء.
5. `sellHops`: مصفوفة من `SwapHop` لعمليات البيع.
6. `minProfitBps`: الحد الأدنى للربح (مثلاً 50 = 0.5%).
7. `deadline`: وقت انتهاء الصلاحية (Unix Timestamp).
8. `nonce`: رقم عشوائي (أو متسلسل) لمنع إعادة تنفيذ نفس الصفقة.
